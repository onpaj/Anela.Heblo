# Knowledge Base: Product Mention Enrichment

**Date:** 2026-03-31
**Branch:** to be created

---

## Problem

KB answers generated by the LLM reference Anela products by name, but the names appear as plain prose. Users cannot easily navigate to a product detail from a KB answer. Czech declension also makes product names inconsistent across answers.

---

## Solution Overview

Two-sided enrichment pipeline:

1. **Input side (`AskQuestionHandler`)** — inject a compact product table into the LLM system prompt and instruct the LLM to replace any product name it mentions with the product's code in brackets, e.g. `(AKL001)`.
2. **Output side (`PostAnswerEnrichmentMiddleware`)** — intercept the LLM response, regex-scan for `(CODE)` patterns, look up canonical name + URL from a cached catalog dictionary, and replace each code annotation with a rich inline reference.

Only `ProductType.Product` and `ProductType.Goods` items are included in the catalog table — raw materials and other types are excluded.

```
AskQuestionHandler
  ├─ ProductEnrichmentCache.GetProductLookupAsync()
  │    (singleton, TTL-cached, resolves ICatalogRepository via IServiceScopeFactory)
  ├─ builds system prompt:
  │    {context} — RAG chunks (existing)
  │    {products} — compact CODE | Název table (new)
  │    {query} — user question (existing)
  │    instruction: replace product names with (CODE)
  └─ _chatClient.GetResponseAsync()
        │
        ▼
   PostAnswerEnrichmentMiddleware
        ├─ forwards to base (Anthropic)
        ├─ LLM returns answer with (CODE) annotations
        ├─ regex scans for (CODE) patterns in catalog dict
        └─ (AKL001) → [Sérum ABC (AKL001)](url)   if URL present
           (AKL001) → Sérum ABC (AKL001)           if URL null
        │
        ▼
   AskQuestionResponse.Answer with inline product references
```

## Components

### 1. `ProductEnrichmentEntry`

Value object holding the data needed for replacement:

```csharp
public class ProductEnrichmentEntry
{
    public string ProductCode { get; set; }
    public string ProductName { get; set; }
    public string? Url { get; set; }  // placeholder — populated in a future story
}
```

---

### 2. `IProductEnrichmentCache` / `ProductEnrichmentCache`

**Responsibility:** provide a fast, cached `CODE → ProductEnrichmentEntry` dictionary for both the handler (prompt building) and the middleware (replacement).

**Registration:** singleton.

**Interface:**
```csharp
public interface IProductEnrichmentCache
{
    Task<IReadOnlyDictionary<string, ProductEnrichmentEntry>> GetProductLookupAsync(CancellationToken ct = default);
}
```

**Implementation:**

- Uses `IServiceScopeFactory` to create a scope and resolve `ICatalogRepository` when a refresh is needed.
- Filters to `ProductType.Product` and `ProductType.Goods`.
- Cache state: a `IReadOnlyDictionary` field + a `DateTime` timestamp, both updated atomically under a `SemaphoreSlim(1,1)` lock on refresh.
- On each call: if `DateTime.UtcNow - _lastLoaded >= TTL` → refresh; otherwise return cached dict.
- TTL: `KnowledgeBaseOptions.ProductEnrichmentCacheTtlMinutes` (default: 60).

---

### 3. `PostAnswerEnrichmentMiddleware` (updated)

Injects `IProductEnrichmentCache`. Overrides `GetResponseAsync`:

1. Forward to `base.GetResponseAsync()` — get LLM response.
2. Load catalog lookup dict via `IProductEnrichmentCache`.
3. Apply regex replacement on `response.Text`:
   - Pattern: `\(([A-Z0-9]+)\)` — matches any `(CODE)` token.
   - For each match: look up `CODE` in dict.
     - Found + URL present → `[Name (CODE)](url)`
     - Found + URL null/empty → `Name (CODE)`
     - Not found → leave match unchanged.
4. Return response with enriched text.

Constructor changes from `(IChatClient inner)` to `(IChatClient inner, IProductEnrichmentCache cache)`.

---

### 4. `AskQuestionHandler` (updated)

Injects `IProductEnrichmentCache`.

Before building the system prompt:
1. Load product lookup dict from cache.
2. Build compact product table string:
   ```
   AKL001 | Sérum s niacinamidem
   KRM002 | Hydratační krém
   ```
3. Replace `{products}` placeholder in the system prompt template.

System prompt template gains `{products}` alongside existing `{context}` and `{query}`. The annotation instruction embedded in the prompt:

> *"Pokud v odpovědi zmiňuješ produkt Anela, nahraď celý název produktu jeho kódem v závorce, např. (AKL001). Použij pouze kódy z přiloženého seznamu produktů."*

If the catalog is empty, `{products}` is replaced with an empty string (no crash, no annotation instruction needed).

## Trade-offs & Constraints

- **Czech declension**: LLM replaces the whole product name with the code, so declension is never an issue in the final output — the canonical name comes from the catalog.
- **Catalog freshness**: singleton cache with 60-min TTL. Catalog changes (new products, renames) propagate within one TTL window. Acceptable given how infrequently the catalog changes.
- **Regex false positives**: `\(([A-Z0-9]+)\)` could match non-product parenthetical tokens. False positives are benign — the code is only replaced if it matches a real catalog entry.
- **URL placeholder**: `ProductEnrichmentEntry.Url` is null until the eshop URL story is implemented. The middleware handles null gracefully.
