# Remove Dead `ErrorMessage` Field from `RunExpeditionListPrintFixResponse` Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Remove the always-null `ErrorMessage` property from `RunExpeditionListPrintFixResponse` and the redundant `Success = true` assignment in its handler, then propagate the resulting type changes through the generated TS client, the hand-coded frontend mirror type, and the hook test mock.

**Architecture:** This is a surgical DTO cleanup. The `BaseResponse` contract already provides structured failure reporting via `ErrorCode` + `Params`, and the `BaseResponse()` constructor defaults `Success` to `true`. We remove the redundant fields, regenerate the OpenAPI TypeScript client (NSwag, runs on `dotnet build` Debug), prune the hand-coded frontend mirror that duplicates the generated shape, and fix the hook test mock. We also add a one-line lock-in xUnit test asserting the constructed response reports `Success == true` without explicit assignment so we cannot accidentally regress the inherited default. The error-fallback branch in `useExpeditionList.ts` that reads `errorData?.errorMessage` from non-OK HTTP bodies is left untouched — it consumes ASP.NET exception middleware payloads, not the typed DTO.

**Tech Stack:** .NET 8, C# 12, xUnit, MediatR, NSwag (OpenAPI client generation), React, TypeScript, Jest, @tanstack/react-query.
