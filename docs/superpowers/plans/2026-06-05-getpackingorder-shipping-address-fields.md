# Expose Shipping Address Fields in GetPackingOrder Response — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Surface the three already-computed shipping address fields (`ShippingStreet`, `ShippingCity`, `ShippingZip`) from the `PackingOrder` domain object through `GET /api/shoptet-orders/{code}/packing` so API consumers can read the delivery address.

**Architecture:** Pure plumbing fix inside a single vertical slice (`Features/ShoptetOrders/UseCases/GetPackingOrder`). Three optional nullable string properties are added to the response DTO and assigned in the handler's existing object initializer. No adapter, persistence, DI, or authorization changes. The generated TypeScript client picks up the new fields automatically on `npm run build`.

**Tech Stack:** .NET 8, MediatR, xUnit + FluentAssertions + Moq, NSwag-generated TypeScript client.

## Background Context for the Engineer

- This codebase is .NET 8 Clean Architecture + Vertical Slice. Each MediatR use case (request, handler, response) lives in its own folder under `Features/<Module>/UseCases/<UseCase>/`.
- **DTOs are classes, never C# records** (project rule, `CLAUDE.md`). NSwag-generated OpenAPI clients mishandle record constructor parameter order. `GetPackingOrderResponse` is already a `class : BaseResponse` — keep the existing pattern.
- The frontend TypeScript API client (`frontend/src/api/generated/api-client.ts`) is generated by NSwag on `npm run build`. The generated file is committed; do not hand-edit it. After backend changes, run `cd frontend && npm run build` so the regenerated diff is committed alongside the C# changes.
- The adapter's `NormalizeAddressField` returns `null` for whitespace-only fields. `null` ⇒ "no address available" semantics. Mirror this end-to-end — never substitute empty strings.
- Tests use xUnit + FluentAssertions + Moq. Pattern: `Arrange → Act → Assert`. Use `.Should().Be(...)` and `.Should().BeNull()`.
