### Question 1
"Errors only" predicate definition. The exact data shape for "has errors" is not specified. Two plausible models: (a) `BankStatement` carries a status enum where one or more values represent an error state, or (b) errors are stored as related records and the predicate is "any related error exists." The implementer should inspect the existing `BankStatement` aggregate / import pipeline and pick the model that matches reality, documenting the choice in the PR. If neither model fits cleanly, escalate before implementing FR-4.

**Answer:** Use model (a). The predicate is `ImportResult != ImportStatus.Success` (i.e., `ImportResult != "OK"`). Implement as a direct EF `Where(bs => bs.ImportResult != "OK")` clause against the `ImportResult` column on `BankStatementImport`. No new related entity or column is introduced.

**Rationale:** `BankStatementImport` (the aggregate persisted in the `BankStatements` table) carries a single `ImportResult` string column whose value is `"OK"` on success and either `PROCESSING_ERROR: …`, `UNKNOWN_ERROR`, or a backend-supplied error message otherwise — see `backend/src/Anela.Heblo.Domain/Features/Bank/ImportStatus.cs` and `ImportBankStatementHandler.cs:89,104,113`. There is no related "errors" table. The existing `BankStatementImportDto.ErrorType` projection and the AutoMapper rule in `BankMappingProfile.cs` already encode "error iff `ImportResult != "OK"`", and the UI in `ImportTab.tsx:205` renders the success badge from `importResult === "OK"`. Reusing this exact predicate keeps semantics consistent across persistence, API, and UI.

### Question 2
Account field location. Confirm whether `Account` is a direct column on `BankStatement` or lives on a related entity. The filter implementation should follow whichever relationship already exists; no new persistence is introduced.

**Answer:** `Account` is a direct column on `BankStatementImport` (the `Account` column on the `BankStatements` table, type `text`, already covered by the existing non-unique index `IX_BankStatements_Account`). Implement the filter as a single case-insensitive `Contains` clause against that column — `query.Where(bs => EF.Functions.ILike(bs.Account, $"%{trimmedAccount}%"))` on PostgreSQL — with no join.

**Rationale:** `BankStatementImport.Account` is a required `string` property mapped directly to a `text` column with index `IX_BankStatements_Account` (see `BankStatementImport.cs:12` and `BankStatementImportConfiguration.cs:39-42,67`). The stored value is the configured account *name* (e.g. `"ShoptetPay-CZK"`, assigned from `accountSetting.Name` at `ImportBankStatementHandler.cs:86`), not an IBAN — which exactly matches the "Účet" column the user sees in the Import-tab grid (`ImportTab.tsx:422-424`). A `Contains`/`ILike` match on this column is what the user expects.

### Question 3
Date type convention. Use whichever date type (`DateOnly` vs. `DateTime`) the existing `StatementDate` and surrounding queries use. Do not introduce a new convention.

**Answer:** Use `DateTime` end-to-end. On the controller boundary expose `dateFrom` and `dateTo` as `string?` query parameters (matching the existing `statementDate`/`importDate` style). Parse them in `GetBankStatementListHandler` via `DateTime.TryParse(...)` into nullable `DateTime?`. Pass to the repository as `DateTime?` and apply inclusive day-granularity bounds against `BankStatementImport.StatementDate` using `.Date >= dateFrom.Value.Date` and `.Date <= dateTo.Value.Date`. Do **not** introduce `DateOnly` into the Bank module.

**Rationale:** `BankStatementImport.StatementDate` is `DateTime` mapped to `timestamp without time zone` (`BankStatementImport.cs:10`, `BankStatementImportConfiguration.cs:29-32`). The existing list endpoint already accepts date filters as `string?` on the wire and parses them with `DateTime.TryParse` in `GetBankStatementListHandler.cs:30-40`, then compares `.Date` equality in `BankStatementImportRepository.cs:31-32`. The frontend already sends ISO date strings from `<input type="date">` controls (`ImportTab.tsx:332-345`). Mirroring the existing string-on-the-wire / `DateTime?` in-handler pattern keeps the OpenAPI surface, the generated TypeScript client, the repository signature, and the EF query all internally consistent — and it avoids creating a one-off `DateOnly` convention in a module that does not otherwise use it.
