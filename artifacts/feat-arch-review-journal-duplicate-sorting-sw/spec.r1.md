# Specification: Extract Duplicated Journal Sorting Logic

## Summary
Eliminate verbatim duplication of the sort-clause `switch` expression in `JournalRepository` by extracting it into a single private static helper method. Both `GetEntriesAsync` and `SearchEntriesAsync` will delegate sorting to the shared helper, removing the KISS/DRY violation without changing any external behavior.

## Background
The architectural review routine flagged that `backend/src/Anela.Heblo.Persistence/Catalog/Journal/JournalRepository.cs` contains the same 12-line sort `switch` expression in two methods (`GetEntriesAsync` lines 44–55 and `SearchEntriesAsync` lines 135–146). The blocks are character-for-character identical. Any future change — a new sortable column, a fix to direction handling, a localization adjustment — must be applied in both places, and one will inevitably be missed. Consolidating the logic improves maintainability and clarifies that the two methods share a single sort contract.

## Functional Requirements

### FR-1: Extract sorting logic into a private static helper
Introduce a private static method on `JournalRepository` with the signature:

```csharp
private static IQueryable<JournalEntry> ApplySorting(
    IQueryable<JournalEntry> query, string? sortBy, string sortDirection)
```

The method must reproduce the existing sort semantics exactly:
- `"title"` (case-insensitive) → sort by `Title`
- `"createdat"` (case-insensitive) → sort by `CreatedAt`
- Any other value, or `null` → sort by `EntryDate` (default)
- `sortDirection == "ASC"` → ascending; any other value → descending (matches current behavior — `"DESC"` and unknown values both fall through to descending)

**Acceptance criteria:**
- `JournalRepository` contains exactly one occurrence of the sort `switch` expression after the change.
- The helper is `private static` and returns `IQueryable<JournalEntry>`.
- Comparison of `sortBy` uses `.ToLower()` to preserve current case-insensitivity behavior.
- Ascending check uses `sortDirection == "ASC"` (exact match, case-sensitive) to preserve current behavior.

### FR-2: Replace both duplicated blocks with a call to the helper
In `GetEntriesAsync` (lines 44–55) and `SearchEntriesAsync` (lines 135–146), replace the duplicated `switch` block with:

```csharp
query = ApplySorting(query, sortBy, sortDirection);
```

**Acceptance criteria:**
- Both methods invoke `ApplySorting` instead of inlining the `switch`.
- The position of the sort step within each method's pipeline (before pagination, after filtering) is preserved.
- No other logic in either method is modified.

### FR-3: Preserve external behavior
The refactor must not alter the SQL generated by EF Core for either method, nor change the API response ordering for any combination of `sortBy` and `sortDirection`.

**Acceptance criteria:**
- Existing unit/integration tests covering Journal listing and search continue to pass without modification.
- For each supported `sortBy` value (`"title"`, `"createdat"`, `null`, unknown) crossed with each `sortDirection` value (`"ASC"`, `"DESC"`, other), the result ordering is identical before and after the change.

### FR-4: Test coverage for the consolidated sort path
Ensure tests exercise the shared helper indirectly through both repository methods. If existing tests already cover the sort matrix for one method but not the other, add the missing cases so a future regression in either caller is caught.

**Acceptance criteria:**
- At least one test per `(sortBy, sortDirection)` combination listed in FR-1 exists for `GetEntriesAsync`.
- At least one test per `(sortBy, sortDirection)` combination listed in FR-1 exists for `SearchEntriesAsync`.
- Tests pass after the refactor.

## Non-Functional Requirements

### NFR-1: Performance
No performance regression. The helper is a static method returning `IQueryable<T>`; EF Core query translation must produce SQL equivalent to the inline version. Verified by inspecting generated SQL in a representative test (e.g., via `query.ToQueryString()`) for at least one sort combination.

### NFR-2: Security
No security implications. The change is a pure internal refactor; no new input is accepted, no authorization paths change.

### NFR-3: Maintainability
After the change, adding a new sortable column requires editing exactly one location. The helper's `switch` expression should remain a single, readable block with one arm per supported column.

## Data Model
No changes. `JournalEntry` entity and its persisted columns (`Title`, `CreatedAt`, `EntryDate`) are unchanged.

## API / Interface Design
No changes to public API. The refactor is entirely internal to `JournalRepository`. The HTTP endpoints backed by `GetEntriesAsync` and `SearchEntriesAsync` retain identical request/response contracts and ordering semantics.

## Dependencies
- `backend/src/Anela.Heblo.Persistence/Catalog/Journal/JournalRepository.cs` — the file being modified.
- `JournalEntry` domain entity (read-only dependency; no changes).
- EF Core's `IQueryable<T>` ordering operators (`OrderBy`, `OrderByDescending`) — already in use.
- Existing Journal repository test suite (location to be confirmed during implementation, typically under `backend/test/Anela.Heblo.Persistence.Tests/` or `backend/test/Anela.Heblo.Tests/`).

## Out of Scope
- Adding new sortable columns (e.g., `UpdatedAt`, `Author`).
- Changing the sort-direction contract from string (`"ASC"`/`"DESC"`) to an enum.
- Making `sortDirection` comparison case-insensitive (current behavior is case-sensitive; preserving it).
- Validating or rejecting unknown `sortBy` values (current behavior silently falls back to `EntryDate`; preserving it).
- Extracting similar sort logic from repositories in other modules.
- Refactoring the surrounding filter/pagination code in either method.

None.
