### Question 1
For `useSaveDashboardSettings`, `useEnableTile`, and `useDisableTile`, the current code uses `await getAuthenticatedApiClient()` (default `showErrorToasts = true`), while `useUserDashboardSettings`/`useTileData` use `getAuthenticatedApiClient(false)` (toasts suppressed). Should the mutation hooks continue showing global error toasts on failure (current behavior, `showErrorToasts` defaults to `true`), or should this be made consistent with the query hooks?

**Answer:** Keep `showErrorToasts` at its default (`true`) for `useSaveDashboardSettings`, `useEnableTile`, and `useDisableTile`. Only remove the redundant `await` on `getAuthenticatedApiClient()` — do not change the boolean argument or add `false`.

**Rationale:** This spec is explicitly scoped to removing the untyped `any`-cast/manual-fetch pattern (see "Out of Scope": no UI/UX changes, no broader behavior cleanup), not to auditing toast UX; changing toast suppression is an unrelated, user-visible behavior change that risks silently hiding mutation failures from users who currently see them, and "surgical changes" project guidance favors leaving it alone absent an explicit request.

### Question 2
The generated `ApiException`/`throwException` error shape (used to detect 403 for the settings/data fallback in FR-2/FR-3) needs to be confirmed exactly (property name for HTTP status on the thrown error) before implementation — this is a quick lookup in `generated/api-client.ts`'s `throwException` function/`ApiException` class, not a product decision, but flagging it so the implementer checks it rather than guessing.

**Answer:** There is no `ApiException` class in this codebase — the generated class is named `SwaggerException` (`frontend/src/api/generated/api-client.ts`, class defined immediately above `throwException`, around line 41216). It has a plain numeric `status` property set directly by its constructor. `throwException(message, status, response, headers, result?)` throws `result` directly if provided (for typed 400/403/404 response bodies elsewhere in the file), but for `dashboard_GetUserSettings` and `dashboard_GetTileData` specifically, every non-200/204 status — including 403 — falls through the generic branch with no `result` argument, so it always throws `new SwaggerException(message, status, response, headers, null)`. The 403 detection code must therefore catch the error and check `error.status === 403` (a plain `number`), referencing `SwaggerException`'s shape, not a nonexistent `ApiException`.

**Rationale:** Verified directly by reading the `SwaggerException` class and `throwException` function in `frontend/src/api/generated/api-client.ts`, and by confirming the specific `processDashboard_GetUserSettings`/`processDashboard_GetTileData` handlers route all non-200/204 statuses (including 403) through the no-`result` branch, which always constructs a `SwaggerException` with `status` set to the real HTTP status code; a grep for `ApiException` across `frontend/src` returns zero matches, confirming the spec's placeholder name was speculative.
