### Question 3
Bulk `GetByIdsAsync` on `ICatalogRepository`. The brief asserts `GetByIdsAsync` is already used by Purchase handlers, but the current handlers call `GetByIdAsync` in loops. Confirm whether a bulk repository method exists; if not, the adapter implements `GetByIdsAsync` by calling `GetAllAsync` once (cached path) or by issuing parallel `GetByIdAsync` calls.

**Answer:** **A real bulk `GetByIdsAsync` method already exists on `ICatalogRepository`** at `backend/src/Anela.Heblo.Domain/Features/Catalog/ICatalogRepository.cs:54` with signature `Task<IReadOnlyDictionary<string, CatalogAggregate>> GetByIdsAsync(IEnumerable<string> ids, CancellationToken cancellationToken = default)` and the explicit comment "Bulk lookup — eliminates N+1 query patterns". The adapter's `GetByIdsAsync` implementation must delegate directly to this repository method (single call), then project each `CatalogAggregate` value to a `MaterialInfo` keyed by `ProductCode`. Do **not** fall back to `GetAllAsync` + filter, and do **not** issue parallel `GetByIdAsync` calls. The brief was correct that the bulk method exists; only the current Purchase handlers fail to use it (which this work fixes as part of FR-4.2 and FR-4.3).

**Rationale:** Evidence is unambiguous — the method is declared, documented, and intended for exactly this use case. Delegating to the existing bulk method is both the simplest implementation and the one that honours NFR-1 (no new round trips, allowed batching improvements).
