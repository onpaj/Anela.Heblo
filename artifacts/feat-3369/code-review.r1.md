### Advisory (cleanup)
- `backend/test/Anela.Heblo.Tests/Architecture/ModuleBoundariesTests.cs:941` — The test is named `Application_types_should_not_catch_SDK_exception_types_directly` but `EnumerateReferencedTypes` explicitly documents it does not inspect method bodies. `catch` clauses are method body constructs, so the test cannot detect a future `catch (MsalException …)` regression — only signature-level leaks (fields, properties, method parameters, return types) are caught. The test name overstates the guarantee. Consider either renaming it to `Application_type_signatures_should_not_reference_SDK_exception_types` or adding an IL-scanning pass (e.g. Mono.Cecil) that walks exception handler tables. Without this, the architectural boundary is enforced only at the signature level, not at the catch-clause level where the original violation lived.
- `backend/src/Adapters/Anela.Heblo.Adapters.Microsoft365/UserManagement/GraphService.cs:168` — Double-logging: `GraphService` logs `LogError` for `MsalException` (line 169–170), then rethrows as `GraphServiceAuthException`; `GetGroupMembersHandler` catches the wrapper and calls `LogError` again (handler line 36). Every auth failure now produces two error log entries for the same fault, the second one less informative since `GraphServiceAuthException.Message` embeds only `msalEx.Message` without `ErrorCode`. The handler's re-log adds no information that isn't already in the adapter log. Consider removing the `LogError` in the handler for these two wrapper types (the adapter already logged with full SDK-level detail), or removing the adapter log and letting the handler log using the wrapper message. Same pattern applies to the `ODataError` / `GraphServiceException` path (adapter line 176, handler line 47).
