### Question 1
Specific `ErrorCode` values for MSAL / OData / Unauthorized exceptions. The codebase likely already has a partial set in `ErrorCodes`. Implementer should reuse existing codes where they fit; if none exist for `ExternalAuthenticationFailed` / `ExternalServiceError` / `Forbidden`, confirm with the team before adding new constants.

**Answer:** Reuse existing codes — do not introduce new ones. The mapping is:
- `MsalException` → `ErrorCodes.ConfigurationError` (0012, HTTP 500). Token acquisition for application permissions failing means the Entra app registration, client secret, or `KeyVault` configuration is broken — it is a server-side configuration problem, not an external service outage.
- `Microsoft.Graph.Models.ODataErrors.ODataError` → `ErrorCodes.ExternalServiceError` (9001, HTTP 503).
- `UnauthorizedAccessException` → `ErrorCodes.Forbidden` (0014, HTTP 403).
- Any other `Exception` → `ErrorCodes.InternalServerError` (0010, HTTP 500), matching the handler's current fallback at `GetGroupMembersHandler.cs:39`.

The spec's reference to a hypothetical `ExternalAuthenticationFailed` constant is dropped. FR-2 should be updated in the next revision to cite these four concrete `ErrorCodes` members verbatim.

**Rationale:** I verified the `ErrorCodes` enum (`backend/src/Anela.Heblo.Application/Shared/ErrorCodes.cs:11-400`) — `Unauthorized` (0013), `Forbidden` (0014), `ExternalServiceError` (9001), `ConfigurationError` (0012), and `InternalServerError` (0010) all already exist with the correct `[HttpStatusCode]` attributes, so no new constants are needed and the surgical-change rule from `CLAUDE.md` is honored. The MSAL→`ConfigurationError` choice diverges from the spec's suggestion but matches the actual semantics: `GraphService.GetGroupMembersAsync` uses `GetAccessTokenForAppAsync` with application permissions (not user context), so an MSAL failure here is always a misconfigured tenant/secret rather than a user-level auth problem — surfacing 503 would mislead an SRE into investigating Microsoft Graph health when the real fix is `kv-heblo-stg`. The Photobank precedent (`GetThumbnailHandler.cs:59-63`) maps `MsalException` to a module-specific 503 code, but Photobank already has dedicated `PhotobankThumbnail*` codes; UserManagement does not, and inventing one solely to mirror that pattern violates YAGNI when `ConfigurationError` already exists. `UnauthorizedAccessException` here is raised when the Entra app lacks the `Group.Read.All` consent — i.e. the caller is authenticated but not permitted, which is exactly the semantic of HTTP 403 / `Forbidden`, not 401 / `Unauthorized`.

### Question 2
Canonical log site. This spec recommends the handler as the single log site (since it has the request context and is the catch-and-convert boundary). Confirm that the service does not have other callers that would lose logging if the service's catches are removed entirely.

**Answer:** Adopt a split policy, not a single canonical site. `GraphService.GetGroupMembersAsync` MUST log at `LogError` immediately before rethrowing each of the four exception types, preserving today's structured fields (`GroupId`, MSAL `ErrorCode`, OData `Error.Code`, scope, request URL). `GetGroupMembersHandler` continues to log at `LogError` once when it catches — but at the handler level the log message is request-scoped ("Failed to handle GetGroupMembers for {GroupId}") and deliberately omits the low-level Graph/MSAL fields the service already captured. Both log sites include the exception object, so the stack trace is emitted at most twice with non-overlapping structured context.

**Rationale:** `IGraphService.GetGroupMembersAsync` has a second production caller — `GraphArticleUserResolver` (`backend/src/Anela.Heblo.Application/Features/UserManagement/Infrastructure/GraphArticleUserResolver.cs:19`) which is invoked by `BackfillArticleRequestedByHandler` (`backend/src/Anela.Heblo.Application/Features/Article/Admin/BackfillArticleRequestedByHandler.cs:37`). That handler has no try/catch at all and was relying on the old swallow-and-return-empty contract; once exceptions propagate, the only place guaranteed to log MSAL `ErrorCode`/OData `Error.Code` for every caller is the service itself. Logging only at the per-use-case handler would mean the Article backfill path loses today's MSAL/OData diagnostics entirely (the article handler doesn't know about Graph-specific fields and shouldn't). NFR-3's "avoid double-logging" intent is preserved by partitioning the structured fields: service owns Graph/MSAL diagnostics, handler owns request-scoped context. The duplicated stack trace is acceptable — the project already logs-and-rethrows in similar boundaries (the global C# style guide at `~/.claude/rules/csharp-coding-style.md` explicitly endorses `LogError(ex, ...); throw;`). FR-2's text should be loosened in the next revision from "logged once, at the handler level" to "logged once at the service (Graph-specific fields) and once at the handler (request-scoped fields)".

### Question 3
Assumption flagged: chose Option A (exceptions propagate, handler catches once). Option B (service returns `Result<T>`) was deferred because it would require a broader project-wide convention change and would touch every consumer of `IGraphService`. If the team has already adopted a Result/discriminated-union convention elsewhere, revisit this choice before implementation.

**Answer:** Confirmed — proceed with Option A. The codebase does not use a `Result<T>`/discriminated-union pattern anywhere; every handler in the repository uses exception-based flow plus a `BaseResponse { Success, ErrorCode }` envelope. Do not introduce `Result<T>` as part of this PR, and do not file a follow-up to convert `IGraphService`. The spec's "Out of Scope" entry covering this stays as written.

**Rationale:** I searched the application layer for any `Result<`/`Either<`/`OneOf<` discriminated-union types and found none — `ErrorCodes.cs`, `BaseResponse`, and the MediatR `IRequestHandler<,>` pattern are the project's only error-flow convention, and they presume exceptions on the throw side and a typed response on the return side. The other consumer of `IGraphService` (`GraphArticleUserResolver` → `BackfillArticleRequestedByHandler`) also follows the exception-based pattern. Option A is therefore the only choice consistent with the surrounding code; introducing `Result<T>` here would create a single-use abstraction at one Graph call site, which violates both YAGNI and the "surgical changes" guidance in `CLAUDE.md`. Revisiting only becomes justified if a future, separate decision document standardizes `Result<T>` repository-wide.
