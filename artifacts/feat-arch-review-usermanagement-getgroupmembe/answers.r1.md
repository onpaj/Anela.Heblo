### Question 1
How does the frontend obtain the `ManufactureGroupId`?

**Answer:** **Option 1 — expose via `ConfigurationController`.** Add a new nullable string property `manufactureGroupId` to `GetConfigurationResponse` (`backend/src/Anela.Heblo.Application/Features/Configuration/GetConfigurationResponse.cs`) and have `GetConfigurationHandler` read it from `IConfiguration["ManufactureGroupId"]` exactly as `ManufactureOrderController` does today. The manufacture frontend reads it from the existing config query (typically wrapped in a `useConfiguration()` / app-config context provider) and passes it into `useResponsiblePersonsQuery(groupId)`. Where the combobox is currently invoked without a way to thread `groupId` through (e.g. `ResponsiblePersonCombobox` is a generic shared component in `frontend/src/components/common/`), keep the manufacture-specific wiring local: have each manufacture caller (`CreateManufactureOrderModal.tsx`, `BasicInfoSection.tsx`, `ManufactureOrderFilters.tsx`) read the configured group ID from the config hook and pass it as a new required `groupId` prop on `ResponsiblePersonCombobox`, which it forwards into `useResponsiblePersonsQuery(groupId)`. Disable the combobox (and skip the query) until `groupId` is non-empty.

**Rationale:** Three reasons make Option 1 the unambiguously correct choice. (a) **Single source of truth stays server-side** — `ManufactureGroupId` is already configured per environment in `appsettings.json` / Azure app-service env vars (`backend/src/Anela.Heblo.API/appsettings.Production.json:37`, `appsettings.json:7`); moving it into a `REACT_APP_*` env var (Option 2) would split the source-of-truth across BE and FE and force a frontend rebuild to change it per environment. (b) **`ConfigurationController` already exists** and is the documented home for exposing safe server-side configuration to the client (`backend/src/Anela.Heblo.API/Controllers/ConfigurationController.cs`); adding one more string property fits the existing pattern with no new endpoint, no new controller, no new module wiring. (c) **Option 3 (a manufacture-side handler that calls `GetGroupMembersHandler` via MediatR) reintroduces the cross-module coupling at the application layer** — the same violation the brief is trying to remove, just shifted one layer deeper. Per `docs/architecture/development_guidelines.md`, modules must not couple across the controller layer; an application-layer handler-to-handler call across module boundaries is explicitly worse than a controller-layer coupling and "likely needs explicit approval." Option 1 is also the spec's own recommendation; no countervailing evidence found in the codebase.

### Question 2
Should `useResponsiblePersonsQuery` be renamed?

**Answer:** **No — keep the name `useResponsiblePersonsQuery`.** Only change its signature (add the `groupId: string` parameter) and its internal URL (now `/api/UserManagement/group-members?groupId={encoded}`). Do not rename the hook, do not rename `ResponsiblePersonCombobox`, do not rename the file. The hook remains in `frontend/src/api/hooks/useUserManagement.ts`.

**Rationale:** Per the project's "Surgical changes" rule in `CLAUDE.md` ("Touch only what the task requires. Don't improve adjacent code... Every changed line should trace directly to the request."), a rename touches four call sites (`ResponsiblePersonCombobox.tsx`, its test file, and any indirect callers via the combobox) plus the underlying file, none of which is required by the architecture finding being fixed. The finding's stated goal is to (a) move the endpoint, (b) remove cross-module imports from `ManufactureOrderController`, and (c) fix logger resolution — none of which require renaming a frontend hook. The hook's name describes its caller-facing intent ("get the people responsible for manufacture orders"), which remains accurate in the manufacture-domain context where it is consumed; only the underlying transport changes. If a future task generalizes the combobox to other domains, that PR can rename then.

### Question 3
Should the MCP tool class be added to MCP documentation?

**Answer:** **Yes — `docs/integrations/mcp-server.md` must be updated in the same PR.** Apply three concrete edits: (1) under the "Manufacture Orders (4)" section, remove the `GetResponsiblePersons` bullet and change the header to **"Manufacture Orders (3)"**; (2) add a new section **"User Management (1)"** containing one bullet: `` `GetGroupMembers` — Entra ID group members by group ID ``; (3) extend the intro phrase "MCP tools for AI assistants to query catalog data, manufacturing orders, and perform batch calculations" to mention user-directory lookups, e.g. "...batch calculations, and user-directory lookups." No other docs sections (Implementation, Tool Pattern, Endpoints, Client Setup) need changes.

**Rationale:** The doc explicitly enumerates every tool by name and groups them by tool-class (Catalog 7, Manufacture Orders 4, Manufacture Batch 4, Knowledge Base 2), so renaming `GetResponsiblePersons` → `GetGroupMembers` and moving it to a new `UserManagementMcpTools` class makes the existing enumeration stale on day one. The brief and spec FR-5 both call for the move; leaving the doc out of date would silently mislead MCP consumers who use this file as the tool inventory. The change is mechanical (three small edits in one file), so bundling it in the same PR has zero risk and avoids the maintenance debt of a deferred doc update.
