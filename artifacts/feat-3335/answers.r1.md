### Question 2
Was commit `745dc72` (merged June 22, 07:50 UTC+2) included in a production Docker image build and deployed before the June 22 nightly run at 02:00 UTC? If yes, the root cause is a different code path not patched by #3243. If no, the deployment gap is the sole cause and FR-3/FR-4 are hardening rather than crash fixes.

**Answer:** No. The deployment gap is confirmed. Commit `745dc72` merged on June 22 at 07:50 UTC+2 (= 05:50 UTC), which is **after** the June 22 nightly run at 02:00 UTC. The June 23 and June 24 runs still produced 14 exceptions each, confirming that no deployment occurred in the roughly 48 hours after the merge. FR-3 and FR-4 are therefore hardening measures (they close latent risks) rather than direct crash fixes; the primary fix is deploying the image built from `745dc72` or any later commit.

**Rationale:** The CI pipeline (`ci-main-branch.yml` lines 415–420) declares `environment: production` on the `deploy-production` job, which enforces the manual approval gate. The telemetry count did not drop on June 23 (it went from 22 to 14, which the spec attributes to a schedule/timezone shift, not a code fix). The arithmetic is unambiguous: 02:00 UTC < 05:50 UTC, so the commit cannot have been included in the image that ran on June 22.

### Question 3
`LedgerItemFlexiDto.LastUpdate` is `DateTime?`. Does the SDK return it with `Kind=Unspecified` (Prague local), `Kind=Local`, or `Kind=Utc`? The current code calls `.ToUniversalTime()` which behaves differently for each Kind. If `Kind=Unspecified`, the server's `TZ` environment variable determines the conversion.

**Answer:** The SDK returns `Kind=Unspecified` representing Prague local time. The correct fix is `TimeZoneInfo.ConvertTimeToUtc(dto.LastUpdate.Value, TimeZoneInfo.Local)` (not `DateTime.SpecifyKind(..., DateTimeKind.Utc)`), which is exactly the pattern already used by `UnspecifiedDateTimeConverter.cs` and `DateTimeLocalKindConverter.cs` in this project. The server's `TZ=Europe/Prague` (set in `Dockerfile`) makes `TimeZoneInfo.Local` resolve to the Prague zone, so the conversion is correct. The current `.ToUniversalTime()` call on `Kind=Unspecified` **accidentally works correctly** on this server because `ToUniversalTime()` on `Unspecified` treats the value as local time — on a container with `TZ=Europe/Prague` that is semantically correct. However it is fragile: if `TZ` is ever unset or changed to UTC, it silently produces wrong timestamps. Replace `.ToUniversalTime()` with `TimeZoneInfo.ConvertTimeToUtc(dto.LastUpdate.Value, TimeZoneInfo.Local)` wrapped in a null-check for consistency with the established project pattern.

**Rationale:** `UnspecifiedDateTimeConverter.cs` (the project's authoritative FlexiBee datetime handling) documents explicitly: "FlexiBee returns dates in local timezone" and applies `DateTime.SpecifyKind(parsed, DateTimeKind.Unspecified)` followed by `TimeZoneInfo.ConvertTimeToUtc(localDateTime, TimeZoneInfo.Local)`. The SDK uses Newtonsoft.Json for deserialisation with no custom `JsonConverter` on the `LastUpdate` property (confirmed by inspecting the `Rem.FlexiBeeSDK.Model.dll` binary — no `JsonConverterAttribute` string is present). Newtonsoft.Json's default behaviour for ISO 8601 strings without a timezone offset (e.g. `"2024-06-15T14:30:00"`) is to return `Kind=Unspecified`. The `Dockerfile` confirms `ENV TZ=Europe/Prague`, so `TimeZoneInfo.Local` is the Prague zone at runtime.
