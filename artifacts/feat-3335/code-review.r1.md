### Advisory (cleanup)
- `backend/src/Adapters/Anela.Heblo.Adapters.Flexi/Analytics/ContactSyncService.cs:147` — `dto.LastUpdate.Value.DateTime` silently discards the offset embedded in the SDK's `DateTimeOffset?`. The assumption that `.DateTime` always yields Prague wall-clock time is correct for the current SDK (FlexiBee returns ISO8601 strings like `"10:00+02:00"`, so `.DateTime` gives `10:00 Unspecified` = Prague local time). However, if the SDK ever returns UTC-valued `DateTimeOffset` (e.g., `"08:00+00:00"`), `.DateTime` gives `08:00 Unspecified`, and `ConvertTimeToUtc` then shifts that by −2 h to `06:00 UTC` — a silent double-shift. The same pattern applies identically in `LedgerSyncService.cs:154`. For these two services the entity property is `DateTimeOffset?`, which Npgsql always accepts regardless of offset, so `dto.LastUpdate.Value.ToUniversalTime()` (the original form) was already safe and semantically unambiguous; the `ConvertTimeToUtc` step here is a consistency change, not a bug fix, and it introduces an offset-stripping assumption. Consider documenting the SDK contract (FlexiBee always returns `DateTimeOffset` with Prague offset, never UTC) in `memory/gotchas/` so future readers understand why discarding the offset is intentional.

- `backend/test/Anela.Heblo.Adapters.Flexi.Tests/Analytics/ContactSyncServiceTests.cs:167` — The regression test for `Map()` is circular: the expected value `TimeZoneInfo.ConvertTimeToUtc(unspecified, TimeZoneInfo.Local)` is computed with the exact same formula the implementation uses, so the test cannot detect a wrong formula — only an accidental deletion of the entire conversion. The same applies to the parallel test in `LedgerSyncServiceTests.cs:274`. A stronger test would hard-code both the input (e.g., `2025-06-19T10:00:00 Unspecified`) and the expected UTC output for a known Prague offset (e.g., `2025-06-19T08:00:00Z` for CEST UTC+2), validating the semantic correctness of the timezone interpretation, not just that the formula was not removed.
