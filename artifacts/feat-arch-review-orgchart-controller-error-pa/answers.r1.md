### Question 1
**Global exception middleware**: Does the project already register a global exception handler that converts unhandled exceptions to a typed `BaseResponse`-shaped 500? If yes, the try/catch should be removed entirely instead of replaced. If no, the typed catch-block replacement is the correct change.

**Answer:** No. The project registers `app.UseExceptionHandler()` in `backend/src/Anela.Heblo.API/Extensions/ApplicationBuilderExtensions.cs:91`, but the only `IExceptionHandler` wired into the chain is `UnauthorizedAccessExceptionHandler` (registered via `services.AddExceptionHandler<UnauthorizedAccessExceptionHandler>()` in `backend/src/Anela.Heblo.API/Extensions/ServiceCollectionExtensions.cs:136`), which only handles `UnauthorizedAccessException` and emits `ProblemDetails`, not `BaseResponse`. No generic handler produces a typed `BaseResponse`/`OrgChartResponse`-shaped 500 for arbitrary unhandled exceptions — the default fallback returns a `ProblemDetails`/empty body. Therefore the typed catch-block replacement inside `OrgChartController.GetOrganizationStructure` is the correct change: keep the existing `try`/`catch` and the `_logger.LogError(ex, ...)` call, and replace only the anonymous-object return with `new OrgChartResponse(ErrorCodes.InternalServerError)`.

**Rationale:** Evidence in the codebase shows a partial exception-handler pipeline that covers `Unauthorized` only. Removing the try/catch would let unhandled `InvalidOperationException`s thrown by `OrgChartService` (e.g. wrapped `HttpRequestException`/`JsonException` from `backend/src/Anela.Heblo.Application/Features/OrgChart/Infrastructure/OrgChartService.cs:59-70`) fall through to a `ProblemDetails` 500 — which is not `OrgChartResponse`-shaped, so it would re-break the contract this spec is trying to fix. A generic `BaseResponse`-aware exception handler is the right long-term direction but is out of scope for this fix and should be tracked as a follow-up.

### Question 2
**Specific error code**: Is there a more specific `ErrorCodes` value (e.g. `ExternalServiceUnavailable`, `UpstreamFailure`) that better describes a SharePoint fetch failure than the generic `ServerError`? If one exists, prefer it; otherwise default to `ServerError`.

**Answer:** Use `ErrorCodes.InternalServerError` (value `0010`). The enum has no `ServerError` member — the brief and spec r1's reference to `ErrorCodes.ServerError` is a naming error. The 500-class candidates in `backend/src/Anela.Heblo.Application/Shared/ErrorCodes.cs` are: `InternalServerError = 0010` (attributed `HttpStatusCode.InternalServerError`), `Exception = 0099` (attributed `HttpStatusCode.InternalServerError`, and reserved by `BaseResponse(Exception)` for the leak-prone "embed exception details" path we are deliberately avoiding), and `ExternalServiceError = 9001` (attributed `HttpStatusCode.ServiceUnavailable` → would silently change the wire status to 503). `InternalServerError` is the only enum value that is generic, attributed to HTTP 500, and not pre-bound to an exception-detail payload. The architect must update spec r1's references to `ErrorCodes.ServerError` (FR-1 acceptance criteria, Data Model section, illustrative code block) to `ErrorCodes.InternalServerError` accordingly.

**Rationale:** `ExternalServiceError` would conflict with the spec's explicit "HTTP status code remains 500" requirement (FR-1) because the project's HTTP status resolution honors the `[HttpStatusCode]` attribute. `Exception` is the code used by `BaseResponse(Exception ex)` (`backend/src/Anela.Heblo.Application/Shared/BaseResponse.cs:44-53`), which stuffs `ex.Message` and `ex.ToString()` into `Params` — the exact leak the spec exists to prevent. The controller's catch is unconditional (it intercepts any exception, not only SharePoint transport faults), so a generic 500 code is the honest description and `InternalServerError` is the only fit.

### Question 3
**Frontend consumers**: Does any existing frontend code currently read `error`/`message` fields from the 500 response? If yes, those call sites need to be updated to read `success`/`errorCode` instead. (A quick repo-wide search for `"Failed to fetch organizational structure"` and consumers of the orgchart endpoint should answer this.)

**Answer:** No. A repo-wide search for `"Failed to fetch organizational structure"` under `frontend/` returns zero hits. The only consumers of the orgchart endpoint are `frontend/src/api/hooks/useOrgChart.ts` (which returns the raw response untouched) and `frontend/src/pages/OrgChartPage.tsx` (which reads `orgChartResponse.organization?.name` and `.positions` on success and renders a generic error banner driven by React Query's `queryError` — the rejection produced by the generated API client on a non-2xx status, not the JSON body). No code path inspects the `error` or `message` fields of the old anonymous shape. No frontend updates are required beyond the automatic regeneration of `OrgChartResponse` via `npm run build`, which adds the typed `success`/`errorCode`/`params` fields for future granular handling.

**Rationale:** The current frontend already treats the 500 as an opaque transport error — React Query rejects on non-2xx and the page renders `(queryError as Error).message` (the HTTP-level message from the fetch layer), so changing the body to a typed `OrgChartResponse` with `success: false`/`errorCode` is a strict superset of today's behavior. Existing UX continues to work unchanged, and the new typed fields become available for future granular error handling without touching any current call site.
