ON CONFLICT ("UserId", "GridKey") DO UPDATE` statement via `ExecuteSqlInterpolatedAsync`. ## PR Summary

Replace the read-then-write `GridLayoutRepository.UpsertAsync` with a single atomic `INSERT ... ON CONFLICT ... DO UPDATE` statement so concurrent debounced column-resize saves no longer race on the unique `(UserId, GridKey)` index. The old pattern issued a `SELECT` followed by an `INSERT` or `UPDATE`; two requests arriving within that window both observed an empty table, both attempted `INSERT`, and the second failed with a unique-constraint violation that surfaced to the user as a silent data loss and flooded the log with false-positive errors.

The fix is a one-method-body change in the persistence layer. Public contract, `DeleteAsync`, migrations, the exception translator, and all frontend code are untouched. A new Postgres integration test class covers the insert path, update path, concurrent race (the bug repro), timestamp control via `FakeTimeProvider`, cancellation, and the translator's non-race error path. The three obsolete in-memory translation tests whose `SaveChangesAsync` seam was bypassed by the raw SQL are removed; equivalent coverage lives in the new integration suite.
