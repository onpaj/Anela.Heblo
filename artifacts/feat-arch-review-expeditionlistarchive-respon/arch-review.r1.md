# Architecture Review: Align ExpeditionListArchive responses with BaseResponse error contract

## Skip Design: true

## Architectural Fit Assessment

The change brings two response classes into line with patterns already enforced everywhere else in the codebase:

- `BaseResponse` (typed `ErrorCode` + `Params`) is the canonical error channel — all 36 other controllers consume it via `BaseApiController.HandleResponse<T>` (`backend/src/Anela.Heblo.API/Controllers/BaseApiController.cs:29`), which already maps `ErrorCode` → HTTP status from the `[HttpStatusCode]` attribute. `ExpeditionListArchiveController` is the outlier — it hand-rolls `BadRequest(response)` / `BadRequest(response.ErrorMessage)`.
- The frontend already has a generated `ErrorCodes` TypeScript enum (`frontend/src/api/generated/api-client.ts:12123`) and idiomatic hooks (`useResetOrderShipment.ts`, `useScanPackingOrder.ts`, `useTransportBoxes.ts:106-107`) consume `data.errorCode` and localize via lookup tables.

Integration points: `ErrorCodes` enum (one new value), `BaseApiController.HandleResponse` (existing helper), `BlobPathValidator` (unchanged), the manual hook in `useExpeditionListArchive.ts` (the only one in this module — the generated client is regenerated by the build but not consumed here). The change is small in scope, low-risk, and removes parallel error machinery.

One spec-level gap: the spec misses that the frontend page-test (`frontend/src/pages/__tests__/ExpeditionListArchivePage.test.tsx:74`) currently mocks `{ success: true, errorMessage: null }` and must be updated in the same PR. Also the spec under-specifies the user-facing failure string — the page displays `err.message` verbatim (`ExpeditionListArchivePage.tsx:109-116`), so throwing the raw `errorCode` would surface "InvalidBlobPath" to the user, which is wrong for a Czech-locale app.

## Proposed Architecture

### Component Overview

```
HTTP client  ─►  ExpeditionListArchiveController
                  │
                  │  Download (file stream on success)         ─►  manual BadRequest(response) on fail
                  │  Reprint  (BaseResponse JSON on success)   ─►  HandleResponse(response)  ◄── base helper
                  ▼
              MediatR ─► DownloadExpeditionListHandler ─► BlobPathValidator ─► IBlobStorageService
                       └─► ReprintExpeditionListHandler ─► BlobPathValidator ─► IBlobStorageService + IPrintQueueSink
                              │
                              └─► returns Response with Success=false, ErrorCode=ErrorCodes.InvalidBlobPath
                                    (via parameterless Fail() factory)

Frontend
  useExpeditionListArchive.ts (manual fetch hook)
    ├── ReprintExpeditionListResponse  { success, errorCode, params }
    ├── Reprint: reads errorCode, maps via REPRINT_ERROR_MESSAGES, throws Error(localized)
    └── Page displays err.message
```

### Key Design Decisions

#### Decision 1: Reprint controller should delegate to `BaseApiController.HandleResponse<T>`

**Options considered:**
- A. Keep the controller's hand-rolled `if (!response.Success) return BadRequest(response); return Ok(response);` (spec's FR-4 implicit choice).
- B. Replace with `return HandleResponse(response);` from the existing base helper.

**Chosen approach:** B for `Reprint`. Keep manual `BadRequest(response)` only for `Download`, where the success path returns a file stream (not `ActionResult<T : BaseResponse>`).

**Rationale:** `HandleResponse` is the established convention across the rest of the codebase. It picks the HTTP status from the `[HttpStatusCode]` attribute on the enum value, logs a warning with `ErrorCode` and `Params`, and returns the correct `ActionResult` type — exactly what is needed and exactly what every other module already does. Hand-rolling `BadRequest(response)` works today but re-introduces a parallel pattern after we just removed one. `Download` cannot use the helper directly because its success body is `application/pdf`, so it keeps manual mapping.

#### Decision 2: Place `InvalidBlobPath` in the FileStorage range (1808)

**Options considered:**
- A. Reuse `ValidationError = 0001` (generic input-validation failure).
- B. Reuse `InvalidUrlFormat = 1801` (existing FileStorage-module member).
- C. Introduce `InvalidBlobPath = 1808` per the spec.

**Chosen approach:** C, matching the spec.

**Rationale:** Two existing FileStorage codes (`InvalidUrlFormat`, `InvalidContainerName`) already differentiate validation failure types at finer granularity than the generic `ValidationError`, so a distinct `InvalidBlobPath` is consistent. `1808` is the next free slot, FileStorage range, `[HttpStatusCode(BadRequest)]` is correct. Tests will assert this specific code, so it must be stable.

#### Decision 3: Frontend hook localizes `errorCode` → Czech string, does not throw the raw code

**Options considered:**
- A. `throw new Error(errorData?.errorCode ?? "HTTP error...")` — spec FR-5 fallback. Surfaces "InvalidBlobPath" to the user.
- B. Map `errorCode` to a localized message inside the hook (pattern from `useResetOrderShipment.ts`, `useScanPackingOrder.ts`), then throw `new Error(localizedMessage)`.

**Chosen approach:** B.

**Rationale:** `ExpeditionListArchivePage.tsx:109-116` renders `err.message` verbatim via `showError("Chyba tisku", msg)`. The current text users see is the server-supplied `"Invalid blob path."`. The spec FR-5 fallback would change this to `"InvalidBlobPath"` — a UX regression. The codebase already establishes a per-hook code → localized-string lookup; replicate that pattern. This stays consistent with `useResetOrderShipment`, requires no new i18n infrastructure, and resolves the spec's Open-Questions concern in line with established convention.

#### Decision 4: `useRunExpeditionListPrintFix` keeps a defensive `errorMessage` fallback

**Options considered:**
- A. Strip all `errorMessage` reading (spec FR-5 literal interpretation).
- B. Keep a defensive fallback because its backing endpoint is on a different, untouched controller.

**Chosen approach:** B for now, with a clear TODO. The endpoint `/api/expedition-list/run-fix` is explicitly out of scope (spec's Out-of-Scope section); we don't know if it returns `errorMessage`, `errorCode`, or something else. Removing the `errorMessage` read without verifying the backend behavior risks regressing this unrelated flow.

**Rationale:** Surgical change — touch only what's required. The hook can read `data.errorCode` first and `data.errorMessage` as a fallback until the matching backend cleanup is filed as a follow-up.

### Interfaces and Contracts

**Backend `Fail()` factories (parameterless):**
```csharp
public static DownloadExpeditionListResponse Fail() =>
    new() { Success = false, ErrorCode = ErrorCodes.InvalidBlobPath };

public static ReprintExpeditionListResponse Fail() =>
    new() { Success = false, ErrorCode = ErrorCodes.InvalidBlobPath };
```

**Backend controller (`Reprint` uses helper, `Download` stays manual):**
```csharp
[HttpPost("reprint")]
public async Task<ActionResult<ReprintExpeditionListResponse>> Reprint([FromBody] ReprintExpeditionListRequest request)
{
    var response = await _mediator.Send(request);
    return HandleResponse(response);
}

// Download keeps manual handling because the success body is binary, not JSON.
if (!response.Success || response.Stream == null)
    return BadRequest(response);  // (no longer response.ErrorMessage)
```

**Frontend types and error mapping:**
```typescript
export interface ReprintExpeditionListResponse {
  success: boolean;
  errorCode: string | null;
  params: Record<string, string> | null;

const REPRINT_ERROR_MESSAGES: Partial<Record<string, string>> = {
  InvalidBlobPath: "Neplatná cesta k souboru.",

// In useReprintExpeditionList queryFn on failure branch:
const errorData = await response.json().catch(() => null);

  (errorData?.errorCode && REPRINT_ERROR_MESSAGES[errorData.errorCode]) ??
  GENERIC_REPRINT_ERROR;
```

For `useRunExpeditionListPrintFix`, keep the fetch shape but prefer `errorCode` with a fallback to `errorMessage` until the underlying controller is also cleaned up:

```typescript
const message =
```

### Data Flow

**Success / Reprint:**
```
POST /api/expedition-list-archive/reprint  { blobPath }
  → Handler validates → blob download → temp file → cups sink
  → Response { Success=true } → HandleResponse → 200 OK { success: true, errorCode: null, params: null }
  → Hook returns data → page shows success toast
```

**Failure / Reprint (invalid blob path):**
```
POST /api/expedition-list-archive/reprint  { blobPath: "../evil" }
  → Handler: BlobPathValidator.IsValid == false → Response.Fail()
  → { Success=false, ErrorCode=InvalidBlobPath }
  → HandleResponse → [HttpStatusCode(BadRequest)] → 400 + JSON body
  → Hook: response.ok == false → reads errorCode → maps to Czech string → throw Error
  → Page: catch → showError("Chyba tisku", "Neplatná cesta k souboru.")
```

**Failure / Download (invalid blob path):**
```
GET /api/expedition-list-archive/download/<bad>
  → Handler: Response.Fail() → controller: BadRequest(response) → 400 + JSON body
  → No JS consumer (frontend uses anchor click via getExpeditionListDownloadUrl) — body purely
     improves OpenAPI consistency, browser still shows a download error.
```

## Specification Amendments

1. **FR-4 refinement:** Use `HandleResponse(response)` from `BaseApiController` in `Reprint` (matches all other controllers). Keep manual `BadRequest(response)` for `Download` because its success path returns a file stream.
2. **FR-5 refinement:** Replace the spec's "throw `new Error(errorData?.errorCode ?? ...)`" fallback with a local `Record<string, string>` map (Czech strings) for `useReprintExpeditionList`, following the pattern in `useResetOrderShipment.ts` and `useScanPackingOrder.ts`. The page displays `err.message` directly, so this protects the user-facing string.
3. **FR-5 refinement:** For `useRunExpeditionListPrintFix`, do not fully strip `errorMessage`. Read `errorCode` first, fall back to `errorMessage`, fall back to HTTP status — its backing controller is out of scope.
4. **FR-6 addition:** Update `frontend/src/pages/__tests__/ExpeditionListArchivePage.test.tsx:74` mock to remove `errorMessage` and add the `BaseResponse` shape.
5. **NFR-2 note:** The frontend hook in this module is hand-rolled (raw `fetch`), not generated from OpenAPI. The OpenAPI-consistency benefit applies to the generated TS client (`frontend/src/api/generated/api-client.ts`), which is rebuilt automatically — verify post-build that the generated `ReprintExpeditionListResponse` no longer contains `errorMessage`.

## Prerequisites

None — no migrations, no config, no infrastructure changes. `ErrorCodes` enum lives in `Anela.Heblo.Application.Shared` and is auto-regenerated into the TS client on `dotnet build`. Existing `BaseApiController.HandleResponse` is already in place. Ship as a single atomic backend+frontend PR (NFR-1).
