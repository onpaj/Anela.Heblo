# Architecture Review: Remove TopProductDto Backward-Compatibility Shims

## Skip Design: true

Backend DTO cleanup with mechanical frontend follow-up. No UI components, layouts, or visual decisions change — the affected screens already render via the canonical `groupKey` / `displayName` properties.

## Architectural Fit Assessment

The change aligns perfectly with the existing architecture and the explicit CLAUDE.md rule against backward-compatibility hacks. `TopProductDto` lives in the canonical contracts location (`Application/Features/Analytics/Contracts/`) and follows the project's DTO-as-class rule (verified — it is a `class`, not a `record`). The handler (`GetProductMarginSummaryHandler.cs`) already populates only `GroupKey` and `DisplayName` — never the shims — so the producer side is already clean. The shims are pure consumption ergonomics that no consumer in this codebase actually depends on (verified by grep across `backend/src`, `backend/test`, and `frontend/src`).

Integration points are minimal and well-isolated:
1. The single DTO file in the Analytics Contracts folder.
2. The NSwag pipeline (auto-regenerates the TS client on `npm run build` / `npm start`).
3. One frontend test fixture that mirrors the DTO shape with legacy field names.

### Key Design Decisions

#### Decision 1: Atomic backend + frontend change in one commit
**Options considered:**
- A) Deprecate first (mark `[Obsolete]`), then remove in a follow-up.
- B) Remove backend properties and frontend usages in a single commit, regenerating the client as part of the change.

**Chosen approach:** B — single atomic commit.
**Rationale:** Solo-developer monorepo with no external API consumers; the OpenAPI client is regenerated on every build. Deprecation phases exist to give external consumers time to migrate — there are none here. A staged removal would itself violate the CLAUDE.md "avoid backwards-compatibility hacks" rule.

#### Decision 2: Keep `"productcode"` / `"productname"` sort-key string aliases in `ApplySorting` (NOT in scope)
**Options considered:**
- A) Remove the legacy `"productcode"` / `"productname"` `switch` arms from `GetProductMarginSummaryHandler.ApplySorting` (lines 172, 175) as part of this change.
- B) Leave them alone — out of scope per the spec.

**Chosen approach:** B — leave them. Flag as **Specification Amendment** below for a follow-up ticket.
**Rationale:** The spec explicitly scopes this to instance-property removal. Sort-key strings are a wire-protocol surface, not a property; removing them risks breaking any cached/persisted frontend sort preference and warrants its own change with its own verification. However, the same backward-compat-shim rule applies, so it should be cleaned up next.

#### Decision 3: No `[JsonPropertyName]` or aliasing tricks to preserve wire compatibility
**Options considered:**
- A) Add `[JsonPropertyName("productCode")]` somewhere to keep the JSON wire field alive.
- B) Remove cleanly, accept that any cached client JSON loses those two fields.

**Chosen approach:** B.
**Rationale:** No external clients exist. The TS client is rebuilt from spec on every FE build. Wire-compat shims would re-introduce the exact pattern we're eliminating.

## Risks and Mitigations

| Risk | Severity | Mitigation |
|---|---|---|
| Hidden consumer of `.ProductCode` / `.ProductName` outside the searched paths (e.g., a Razor view, integration test fixture, or seeded data file). | Low | Re-run a case-insensitive grep across the *entire* repo (not just `backend/src` + `frontend/src`) before committing: `grep -rni "\.productCode\|\.productName\|\.ProductCode\|\.ProductName"` filtered to TopProductDto context. The spec's grep guidance is correct but narrower than full-repo. |
| Stale generated client checked in to git after backend change but before FE regenerates. | Low | Commit must include both the backend edit *and* the regenerated `api-client.ts`. Run `npm run generate-client` locally before `git add`. |
| Frontend caching of a sort preference still using `"productCode"` / `"productName"` string. | Low | Sort-key aliases in `ApplySorting` remain (Decision 2), so cached preferences keep working. Verify the FE does not currently emit these strings — quick scan of `useProductMarginSummary.ts` and `ProductMarginSummary.tsx`. |
| Test fixture edit accidentally touches `MonthlyProductSegmentDto` fields (lines 36–37, 49–50). | Low | Edit only lines 67–68 in `ProductMarginSummary.test.tsx`. Reviewer must confirm only the `topProducts` array changes, not the `monthlyData` array. |
| NSwag regeneration drift (unrelated diffs creeping into `api-client.ts`). | Medium | Before committing, `git diff frontend/src/api/generated/api-client.ts` should show only deletions/changes within the `TopProductDto` class and `ITopProductDto` interface. Any other diff indicates a stale base — rebuild before committing. |

## Specification Amendments

1. **Add explicit step: regenerate the TS client.** FR-3's acceptance criterion needs the explicit command from `docs/development/api-client-generation.md`: run `npm run generate-client` (or `npm run build`) from `frontend/`. This is implied by FR-3 but should be written so the executing agent does not skip it and commit a stale `api-client.ts`.

2. **Add file-list pre-check.** Before edits, the executing agent should run `grep -rni "ProductCode\|ProductName" <repo>` filtered to lines mentioning `TopProductDto` context (or to the three known files only). This narrows blast radius and catches any consumer outside the spec's three-file mental model.

3. **Clarify test-fixture scope.** FR-4's acceptance criterion should explicitly call out that **only** the `topProducts` fixture entries in `ProductMarginSummary.test.tsx` change — `MonthlyProductSegmentDto` entries genuinely have `productCode`/`productName` fields and must remain untouched. This is the most likely place a hasty find-and-replace would introduce a regression.

4. **File a follow-up ticket (out-of-scope cleanup).** `GetProductMarginSummaryHandler.ApplySorting` (lines 172, 175) accepts `"productcode"` and `"productname"` as sort-key string aliases. These are the wire-protocol equivalent of the same backward-compat shim and should be removed in a separate change after this one merges — but only after verifying no FE caller still emits those strings.
