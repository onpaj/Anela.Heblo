# Architecture Review: Extract Shared MarketingFolderLinkRequest DTO

## Skip Design: true

This is a pure backend type-level refactor. No new UI components, no visual changes, no UX flow alterations. The frontend's hand-written marketing components (`MarketingActionModal.tsx`, `MarketingActionGrid.tsx`, `useMarketingCalendar.ts`) use locally-defined `FolderLinkInput` interfaces and inline structural shapes — they do not import the generated `CreateFolderLinkRequest` class. After regeneration the generated `api-client.ts` will rename the class transparently; no design work required.

## Architectural Fit Assessment

The proposal aligns cleanly with the codebase's existing conventions:

- **Filesystem layout** — `docs/architecture/development_guidelines.md` (§ Contracts and DTOs Rules, line 7+) states "DTO objects for API (`Request`, `Response`) live in `contracts/` of the specific module". The Marketing Contracts folder already follows a strict "one type per file" pattern (`CreateMarketingActionRequest.cs`, `UpdateMarketingActionRequest.cs`, `MarketingActionFolderLinkDto.cs`, etc.). The proposed `MarketingFolderLinkRequest.cs` slots in without disruption.
- **DTO-as-class rule** — `CLAUDE.md` and `development_guidelines.md` mandate classes (not records) for OpenAPI-bound DTOs. Spec FR-1 already binds this.
- **Naming** — `MarketingActionFolderLinkDto` (response side) already exists in the same folder. The new request-side counterpart should naturally be `MarketingActionFolderLinkRequest` to mirror it exactly. **Spec chose `MarketingFolderLinkRequest`** (without `Action`). Both are defensible; see Amendments below.
- **No module-boundary concerns** — both `CreateMarketingActionRequest` and `UpdateMarketingActionRequest` live in the same module/folder, so introducing a sibling shared DTO does not introduce any cross-module dependency.

**Integration points (confirmed by grep):** the rename surfaces in exactly four backend locations and the regenerated FE client.

## Proposed Architecture

### Component Overview

```
backend/src/Anela.Heblo.Application/Features/Marketing/Contracts/
├── CreateMarketingActionRequest.cs        ──┐
│       FolderLinks: List<MarketingFolderLinkRequest>?
├── UpdateMarketingActionRequest.cs        ──┤
│       FolderLinks: List<MarketingFolderLinkRequest>?
└── MarketingFolderLinkRequest.cs   (new)  ◄─┘
        FolderKey: string (Required, MaxLength 100)
        FolderType: MarketingFolderType (Required)

         │ generated on build (NSwag)
         ▼
frontend/src/api/generated/api-client.ts
    class MarketingFolderLinkRequest      (renamed from CreateFolderLinkRequest)
    CreateMarketingActionRequestData.folderLinks: MarketingFolderLinkRequest[]
    UpdateMarketingActionRequest.folderLinks:     MarketingFolderLinkRequest[]
```

### Key Design Decisions

#### Decision 1: Co-locate the shared DTO as a sibling file in the same Contracts folder
**Options considered:**
1. Sibling file in `Marketing/Contracts/MarketingFolderLinkRequest.cs` (spec proposal).
2. Sub-folder `Marketing/Contracts/Shared/` for cross-use-case types.
3. Keep nested but in `Update` instead — just inverting the coupling.

**Chosen approach:** Option 1 (sibling file). Matches the existing flat layout (`MarketingActionFolderLinkDto.cs` is already a flat sibling). Introducing a `Shared/` subfolder for a single class is YAGNI.

**Rationale:** Consistent with the convention already established by `MarketingActionFolderLinkDto.cs`. Option 3 just relocates the coupling.

#### Decision 2: Plain class with public mutable setters, not a `record` or `init`-only properties
**Options considered:**
1. `public class` with `{ get; set; }` (matches the existing nested type and all sibling Requests in the folder).
2. `record` or `init`-only — more idiomatic .NET but **forbidden** by the project rule.

**Chosen approach:** Option 1.

**Rationale:** Binding project rule — OpenAPI client generators mishandle record parameter order. The existing nested class uses `{ get; set; }`; the extracted class must preserve that exact shape for wire-format compatibility (FR-6).

#### Decision 3: Do not introduce an internal "Shared/Common" contracts assembly
**Options considered:**
1. Keep within the Marketing module's `Contracts/` folder.
2. Move to a cross-module shared contracts location.

**Chosen approach:** Option 1.

**Rationale:** Per `development_guidelines.md` line 11: "DTOs are **never shared or global**" (i.e., across modules). Within a single module, sharing across use cases is fine. Both contracts that consume this type live in the Marketing module.

#### Decision 4: Preserve nullability of the `FolderLinks` collection on each request unchanged
Both `CreateMarketingActionRequest.FolderLinks` (line 28) and `UpdateMarketingActionRequest.FolderLinks` (line 30) are currently `List<...>?` (nullable). Retain `List<MarketingFolderLinkRequest>?` on both. Do not "tidy up" by harmonising other attributes — the spec is type-only.

No data-flow change. ## Specification Amendments

1. **FR-5 acceptance — clarify the "zero matches" expectation.** The phrase "A repository-wide search under `frontend/src/` for `CreateFolderLinkRequest` returns zero matches" should be qualified with **"after running the build/generator step that refreshes `frontend/src/api/generated/api-client.ts`."** Otherwise the check trivially fails until generation runs. Suggest rewording the AC to: *"After running the build, a search under `frontend/src/` for `CreateFolderLinkRequest` (including generated code) returns zero matches."*

2. **FR-6 acceptance — drop "ordering hints".** "Property names, casing, ordering hints, and nested object shape stay identical" — JSON object property order is not part of the JSON data model and is not enforced by either client. Recommend striking "ordering hints" to avoid an unenforceable AC. Property order in the generated TS class constructor and `toJSON` is implementation detail of NSwag.

3. **Naming alternative for consideration (non-binding rename to spec).** Sibling response DTO is `MarketingActionFolderLinkDto`. For perfect symmetry the new request could be `MarketingActionFolderLinkRequest`. Spec binds `MarketingFolderLinkRequest`; that is acceptable and shorter. If the spec author wants exact mirroring, this is the only opportunity to switch — flagging here. Otherwise proceed with the spec name.

4. **Explicit non-change for handlers.** Spec FR-4 lists "Create handler, Update handler" among consumers. Verification shows both handlers reference only `request.FolderLinks` and `link.FolderKey` / `link.FolderType` — **no type name** is mentioned. Recommend FR-4's AC be reworded so a developer doesn't try to "update" handler files that contain nothing to update.

5. **Explicit non-change for frontend hand-written code.** Spec FR-5 implies "Any frontend code currently importing or using the old `CreateFolderLinkRequest` type must be updated." Verification: no non-generated FE file imports or names `CreateFolderLinkRequest`. The FE uses local types (`FolderLinkInput`) and inline `Array<{ folderKey; folderType }>`. The PR should expect **zero hand-written FE diffs**.
