### Question 1
How should `DocumentType` be replaced in the ingestion job?

**Answer:** Choose **Option B**: move the `DocumentType` enum out of `Anela.Heblo.Domain.Features.KnowledgeBase` and into `Anela.Heblo.Application.Shared.Rag` (alongside the existing `OneDriveFolderMapping`). Update all current consumers — `KnowledgeBaseDocument`, `KnowledgeBaseChunk`, `OneDriveFolderMapping`, the EF Core configuration/migrations metadata, and `LeafletIngestionJob` — to reference the new shared location via `using Anela.Heblo.Application.Shared.Rag;`. No enum members change; the underlying int values stay identical so the existing database column remains compatible with no migration required. This explicitly supersedes the spec's Out of Scope clause "Removing or refactoring `DocumentType` inside the KnowledgeBase module — it stays there for KnowledgeBase's own use" — the FR-4 goal cannot be reached without relocating the type.

**Rationale:** `DocumentType` is already a cross-module concept in practice — its four members (`KnowledgeBase`, `Conversation`, `Leaflet`, `Article`) span four different feature modules, and the shared `Application/Shared/Rag/OneDriveFolderMapping.cs` already takes a hard dependency on it. Option A (a Leaflet-owned duplicate enum) does not compile cleanly because `OneDriveFolderMapping.DocumentType` is typed as the KB enum, so the Leaflet job's `.Where(m => m.DocumentType == LeafletDocumentType.Leaflet)` would be a type mismatch — making Option A only viable if `OneDriveFolderMapping` itself is also restructured, which is more invasive than just moving the enum. Option C does not fit because the mappings are not KB-owned (see Q2). Moving `DocumentType` to `Application/Shared/Rag/` mirrors the existing shared-RAG-infrastructure pattern (`OneDriveFolderMapping`, `RagFeatureOptions`, `RagQueryExpansionConfig`) and removes the KnowledgeBase-domain dependency from every module that reads or filters by it.

### Question 2
Is `OneDriveFolderMappings` a KnowledgeBase-owned type?

**Answer:** **No.** `OneDriveFolderMapping` lives in `backend/src/Anela.Heblo.Application/Shared/Rag/OneDriveFolderMapping.cs` and the `OneDriveFolderMappings` collection is declared on the shared base class `RagFeatureOptions` (`Application/Shared/Rag/RagFeatureOptions.cs`), which `LeafletOptions` inherits from. The Leaflet ingestion job continues to read mappings directly from `LeafletOptions.OneDriveFolderMappings` (its own inherited config), so no `GetMappingsAsync(filter)` method on `ILeafletKnowledgeSource` is required and Option C is not adopted.

**Rationale:** Each module binds its own `RagFeatureOptions`-derived options class to its own configuration section (`Leaflet:OneDriveFolderMappings`, `KnowledgeBase:OneDriveFolderMappings`), and `KnowledgeBaseIngestionJob` already iterates `_options.OneDriveFolderMappings` without any `DocumentType` filter — confirming the convention that each module's options contain only that module's mappings. The Leaflet job's current `.Where(m => m.DocumentType == DocumentType.Leaflet)` filter is therefore defensive, not load-bearing. Once `DocumentType` moves to `Application/Shared/Rag/` (Q1), the filter still works against the Leaflet-section mappings without any cross-module domain reference.

### Question 3
Architecture-test framework choice — NetArchTest vs. hand-rolled reflection test vs. csproj/Analyzer-level enforcement.

**Answer:** Use a **hand-rolled xUnit reflection test** in the existing `Anela.Heblo.Tests` project. Add a new file `Architecture/ModuleBoundariesTests.cs` containing a `[Fact]` (or `[Theory]` over (consumer, forbidden-namespace) pairs) that loads the `Anela.Heblo.Application` assembly via `Assembly.Load("Anela.Heblo.Application")`, enumerates all types whose namespace starts with `Anela.Heblo.Application.Features.Leaflet`, and asserts that none of their referenced types (constructor parameters, fields, properties, method parameters, return types, generic arguments, attributes) live in the `Anela.Heblo.Domain.Features.KnowledgeBase` namespace. Include a curated allowlist (initially empty) so future intentional exceptions can be made explicitly. Do **not** add the NetArchTest NuGet package, and do **not** introduce a Roslyn analyzer or csproj-level reference rule.

**Rationale:** The repository does not currently depend on NetArchTest, and an existing pattern for reflection-driven cross-cutting tests already lives in the same project (`backend/test/Anela.Heblo.Tests/ReflectionValidationTests.cs`). A 30–60 line hand-rolled test reuses that pattern, adds zero new dependencies, and produces a clear failure message identifying the offending type and member — which a generic NetArchTest "fluent" assertion would not. A Roslyn analyzer or csproj reference rule would block the entire `Anela.Heblo.Application` assembly from referencing the KB domain, which is too broad — KnowledgeBase's own application-layer code legitimately needs that reference. A reflection test scoped to the Leaflet namespace expresses the actual rule precisely.
