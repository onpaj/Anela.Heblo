### Question 1
Behavior when `CurrentUser.Id` is null in `SetCarrierCoolingHandler` / `SetGiftSettingHandler`. Today, controllers throw `UnauthorizedAccessException` from `GetCurrentUserId()` when no claim is present. After migration, handlers obtain `CurrentUser.Id`, which can be null. Options: (A) typed failure (`Success = false`, `ErrorCode = ErrorCodes.Unauthorized`); (B) throw `UnauthorizedAccessException`; (C) use `CurrentUserExtensions.GetIdentifier()` which falls back to `"system"`.

**Answer:** Option (A). Both handlers must return a typed failure response with `Success = false` and `ErrorCode = ErrorCodes.Unauthorized` when `CurrentUser.Id` is null or empty. Do not throw `UnauthorizedAccessException`, and do not fall back to `"system"` via `CurrentUserExtensions.GetIdentifier()`.

**Rationale:** The established convention for handlers that require an identified caller is the typed-failure pattern — `CreateMarketingActionHandler` (`backend/src/Anela.Heblo.Application/Features/Marketing/UseCases/CreateMarketingAction/CreateMarketingActionHandler.cs:40-45`) checks `IsAuthenticated`/`Id` and returns a domain-specific `Unauthorized*` error code, and the same shape is used across Marketing and Journal handlers. The shared `ErrorCodes.Unauthorized` (value 0013, declared at `backend/src/Anela.Heblo.Application/Shared/ErrorCodes.cs:38-39`) is already mapped to HTTP 401 by the `HttpStatusCode` attribute, so this preserves the existing observable behavior (`UnauthorizedAccessException` → 401) while remaining idiomatic. Falling back to `"system"` would silently attribute writes to an unidentified caller — unacceptable for `ModifiedBy` audit columns.

### Question 2
Should we introduce `ICurrentUserService.GetRequiredUserId()` as a convenience? Seven handlers will now write the same `var userId = _currentUser.GetCurrentUser().Id ?? throw …` pattern.

**Answer:** No. Do not add `GetRequiredUserId()` (or any other new method) to `ICurrentUserService` in this change. Each migrated handler inlines `var userId = _currentUserService.GetCurrentUser().Id;` and applies its own null-handling — the `"anonymous"` defense-in-depth fallback for the five Dashboard handlers per FR-2, and the `ErrorCodes.Unauthorized` typed failure for `SetCarrierCoolingHandler` / `SetGiftSettingHandler` per Q1.

**Rationale:** A single `GetRequiredUserId()` cannot serve the three distinct null-handling policies that already coexist in this codebase (`"anonymous"` defense-in-depth, typed `Unauthorized` failure, `GetIdentifier()` → `"system"`), so a convenience method would either pick one and force the others to bypass it, or balloon into multiple overloads. Adding it also conflicts with the surgical-change guidance in `CLAUDE.md` and the YAGNI rule — seven inline call sites are not painful enough to justify an interface change. Revisit only if a follow-up adds a fourth call site that shares one of the existing policies verbatim.

### Question 3
Frontend impact surface. Confirm via grep that the only places sending `userId` / `modifiedBy` in those request bodies are the auto-generated client + the hooks that call it; if any handcrafted code sets these fields, it must be updated as part of FR-5.

**Answer:** Confirmed — no handcrafted frontend code currently sends `userId` or `modifiedBy` in the request body for any of the affected endpoints. `frontend/src/api/hooks/useDashboard.ts` has zero references to `userId`/`modifiedBy`; `frontend/src/api/hooks/useCarrierCooling.ts` has zero references; `frontend/src/api/hooks/useGiftSetting.ts` references `modifiedBy` only as a read-only field on the response `GiftSettingDto` (its `SetGiftSettingRequest` already omits it). All other `userId:` / `modifiedBy:` matches in `frontend/src` belong to unrelated features (feedback, stock-taking, knowledge-base, leaflet, journal). The implementer must still re-run `grep -rn "userId\|modifiedBy" frontend/src` after regenerating the OpenAPI client and fix any TypeScript compile errors introduced by the regenerated request DTO types — but no handcrafted-code edits are expected.

**Rationale:** A direct grep of `frontend/src` against the three affected hook files shows the contract change is confined to the generated `frontend/src/api/generated/api-client.ts` and the seven backend endpoints. FR-5's residual risk surface is therefore the regenerated DTOs themselves, not consumer code, and the pre-merge grep + `npm run build` already mandated by FR-5 are sufficient verification.
