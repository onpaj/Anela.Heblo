### Question 1
Contract namespace location. The two candidate homes for the relocated `IPrintQueueSink` interface are (a) a new `Anela.Heblo.Application.Shared.Printing` namespace, (b) a `Contracts/` subfolder under whichever module owns it conceptually (`ExpeditionList` if it is the producer, `ExpeditionListArchive` if treated as a consumer-defined contract per the `ILeafletKnowledgeSource` pattern). The `ILeafletKnowledgeSource` exemplar suggests the **consumer** defines the contract, which would mean `ExpeditionListArchive` owns `IPrintQueueSink`. But there are multiple consumers (both `ExpeditionList` and `ExpeditionListArchive` print), so a shared location may be more honest. Architect to decide before implementation begins.

**Answer:** Move `IPrintQueueSink` to `Anela.Heblo.Application.Shared.Printing.IPrintQueueSink`, in a new file at `backend/src/Anela.Heblo.Application/Shared/Printing/IPrintQueueSink.cs`. The interface signature (`Task SendAsync(IEnumerable<string> filePaths, CancellationToken cancellationToken = default)`) is unchanged. Update the two consumers (`ExpeditionListService` in `ExpeditionList`; `ReprintExpeditionListHandler` and `ExpeditionListArchiveModule` in `ExpeditionListArchive`) and all four implementations (`FileSystemPrintQueueSink`, `CombinedPrintQueueSink`, `CupsPrintQueueSink`, `AzureBlobPrintQueueSink`) plus the DI registrations in `Anela.Heblo.API/Extensions/ServiceCollectionExtensions.cs:397–410` and `Anela.Heblo.Adapters.Cups/CupsAdapterServiceCollectionExtensions.cs:31` to reference the new namespace.

**Rationale:** The `ILeafletKnowledgeSource` "consumer-owned contract" exemplar fits a single-consumer / single-provider relationship — but `IPrintQueueSink` has **two genuine consumers in `Application`** (`ExpeditionListService` in `ExpeditionList` and `ReprintExpeditionListHandler` in `ExpeditionListArchive`, verified via `grep -rn "IPrintQueueSink" backend/src`) and **three independent provider adapters** (`Anela.Heblo.Adapters.Cups`, `Anela.Heblo.Adapters.Azure`, and the `FileSystemPrintQueueSink` / `CombinedPrintQueueSink` in the Application/API projects). Pushing the contract into either consumer's `Contracts/` folder would just flip the cross-module import the other way. `Anela.Heblo.Application.Shared` is already established in the codebase as a namespace inside the existing `Application` project (it hosts `BaseResponse`, `ErrorCodes`, `Json`, `Rag`, `WebSearch`, etc., consumed by many feature modules) — it is **not** a separate "Shared" csproj that the development guidelines (Common Pitfall #4) warn against. A new `Shared/Printing` subfolder is the cleanest, lowest-friction home and lets the `ModuleBoundariesTests` reflection check be extended to enforce that neither `ExpeditionListArchive` nor `ExpeditionList` references the other.

### Question 2
Configuration migration. Should `ExpeditionListArchiveOptions.BlobContainerName` be sourced from a new config section, or should the Archive module bind from the existing `ExpeditionList:BlobContainerName` value to avoid env-config churn? Recommended default: new section (`ExpeditionListArchive:BlobContainerName`) for cleanliness, with deployment instructions to copy the value over. The default `"expedition-lists"` covers environments where the section is missing.

**Answer:** Bind `ExpeditionListArchiveOptions` to the **existing** `ExpeditionList` configuration section — i.e. `services.Configure<ExpeditionListArchiveOptions>(configuration.GetSection("ExpeditionList"))` inside `ExpeditionListArchiveModule.AddExpeditionListArchiveModule(...)` (the method's signature must change to accept `IConfiguration`, matching how `ExpeditionListModule.AddExpeditionListModule` already does it). Do **not** introduce a new `ExpeditionListArchive` configuration section. The Archive's options class exposes a single property `string BlobContainerName { get; set; } = "expedition-lists";` and reads the same `BlobContainerName` key that `Anela.Heblo.API/appsettings.{Development,Staging,Production,Test}.json` and `appsettings.json` already populate under `ExpeditionList`. No appsettings*.json, Azure App Service application settings, Conductor config, or deployment scripts need to change as part of this refactor.

**Rationale:** The blob container is a **single physical Azure resource shared by both modules** — `AzureAdapterModule` constructs a singleton `BlobContainerClient` from `PrintPickingListOptions.BlobConnectionString` + `.BlobContainerName` (`Adapters.Azure/AzureAdapterModule.cs:18–22`), so the Archive's reads and the producer's writes already hit the same container. A separate `ExpeditionListArchive:BlobContainerName` section would have to be kept in lockstep across five environment-specific config files (Development uses `"expedition-lists-stg"`, Production/Staging/Test/appsettings.json each have their own values) — drift between the two sections in production would silently break reprint/download. .NET's options system supports binding multiple typed views on the same section, which is precisely what this case calls for. The Archive still owns its own options **type** (which eliminates the compile-time dependency on `PrintPickingListOptions`); it does not need to own its own configuration section. Note: three other Archive handlers (`DownloadExpeditionListHandler`, `GetExpeditionListsByDateHandler`, `GetExpeditionDatesHandler`) also consume `IOptions<PrintPickingListOptions>` for `BlobContainerName` — they must be rewired to `IOptions<ExpeditionListArchiveOptions>` in the same change so the Archive module is genuinely free of `ExpeditionList` imports.

### Question 3
`ExpeditionListController` scope. This spec creates the controller solely to host `RunFix`. Should other `ExpeditionList` operations (currently in `Anela.Heblo.API/Features/ExpeditionList/`) also be migrated into the new controller as part of this change, or is that scoped out? Default assumption: scoped out — only `RunFix` moves. Architect can override if there's an obvious adjacent cleanup.

**Answer:** Scoped out. The new `ExpeditionListController` exposes exactly one action: `RunFix` at `POST /api/expedition-list/run-fix`. No other endpoints are migrated, renamed, or added in this change.

**Rationale:** A directory scan of `backend/src/Anela.Heblo.API/Controllers/` and `backend/src/Anela.Heblo.API/Features/ExpeditionList/` confirms there are **no other ExpeditionList HTTP endpoints anywhere in the API project** — the single file under `API/Features/ExpeditionList/` is `CombinedPrintQueueSink.cs`, a DI adapter (not an MVC controller action). There is no adjacent work to consolidate. Keeping the controller minimal honors the brief ("just move the one action") and the project's "Surgical changes" rule in `CLAUDE.md`. Future ExpeditionList endpoints can be added to this controller as they are written.
